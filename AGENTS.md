# StyleLens 开发总纲

> 适用范围：本仓库所有代码、文档、模板和脚本。
> 优先级：用户最新明确指令最高；本文件高于 README、docs 和历史聊天记录。

## 1. 当前统一口径

1. 正式开发目录固定为：
   - `/Users/shaobaolu/Downloads/AI 工程/Vibe coding 项目/StyleLens`
2. 可信基线是 `origin/main`。
3. 开始新任务前，必须确认当前工作是否明确基于 `origin/main`。
4. 后续所有开发默认只允许在正式开发目录的 `main` 分支上进行；禁止默认创建 `codex/...` 或其他功能分支。
5. 只有用户明确要求开分支时，才允许创建分支，并必须说明分支名、基线和合并目标。
6. 历史 worktree 只允许只读取证、差异比对、确认遗留任务状态；不得作为正式开发、验收、上传或发布目录。
7. README 只作为项目背景说明；涉及开发目录、可信基线、Git、验证、发布流程时，以本文件为准。

## 2. 新任务开工检查

每个新任务开始前，先执行并阅读：

```bash
git fetch origin --prune
git status --short --branch
git branch --format='%(refname:short)|%(upstream:short)|%(objectname:short)|%(worktreepath)'
git worktree list --porcelain
```

必须确认：

1. 当前工作区是否干净。
2. 当前分支是否基于最新 `origin/main`。
3. 是否有未提交改动、未跟踪文件、历史 worktree 或未收口分支。
4. 如果现有改动可能和本任务冲突，先提示用户，不得擅自覆盖。
5. 未经用户明确要求，不得自动删除分支、删除 worktree、强制 reset 或回滚他人改动。

## 3. 修改边界

1. 只做用户明确要求的事，不擅自加功能。
2. 小问题小改，禁止顺手大重构。
3. 删除核心文件、大范围重构、改变核心业务逻辑前，必须先征得用户同意。
4. 不确定时先查代码和文档，不靠猜。
5. 不回滚用户或其他线程的改动，除非用户明确要求。

## 4. 安全红线

1. 不硬编码密码、Token、API Key、数据库密钥或其他敏感信息。
2. 敏感配置只能放在 `.env`、Vercel 环境变量或对应平台安全配置中。
3. 不在回复、文档或提交信息里输出完整密钥、Token、数据库密码或连接串。
4. 本地预览 / 开发环境默认不得直连生产真实数据，除非用户明确要求并说明风险。

## 5. 验证规则

代码任务完成前，至少按影响范围选择验证：

```bash
npm test
npx tsc --noEmit
npm run build
```

补充规则：

1. 涉及单个模块时，先跑相关测试，再跑必要的全量检查。
2. 涉及 UI 的浏览器、截图、视觉验收，只有用户明确同意才执行。
3. 如果全量检查失败，必须区分是本次改动导致，还是项目既有问题。
4. 不得在未验证的情况下宣称“完成”“已修复”“可以上线”。

## 6. Git 与发布

1. 任务实现、验证、问题收敛后，必须先询问用户是否提交、push 或合并。
2. 未经用户明确确认，不得自动提交、push、合并或创建 PR。
3. 如果当前分支直接跟踪 `origin/main`，提交前必须说明“这是提交并推送到 main”。
4. 提交和 push 默认提交到 `main`；如果当前不在 `main`，必须先说明并征得用户确认，不得直接 push 功能分支。
5. 提交前必须再次检查工作区状态，只暂存本次任务相关文件。
6. 提交信息要说明真实改动，不写空泛的“update”“fix”。

## 7. StyleLens 专属约束

1. 保留 URL / 图片风格提取主流程，除非用户明确要求重构。
2. Publisher 当前必须完整支持 8 个官网模板和 15 个 Dashboard 模板：文本 / `.md` / `.txt` 生成官网 HTML，`.csv` / `.json` / `.xlsx` 数据文件生成 Dashboard HTML。PPT、Skill、GitHub 开源仍不混入当前版本。
3. `.claude/worktrees`、`.next`、`node_modules`、本地缓存和历史工作区不能作为正式源码依据。
4. 根目录素材、截图、缓存文件需要谨慎处理；测试素材应归档到 `test/fixtures/`，历史设计稿或一次性调试产物应归档到 `docs/archive/`。
5. `walkthrough.md` 是外部流程维护文件，普通开发和重构中不要随意修改。

## 8. 报错处理

1. 连续报错超过 2 次，必须停下来汇报现状。
2. 不知道原因时直接说明，不胡编不存在的包、接口或文档。
3. 依赖安装失败、接口不存在、文档缺失或指定方案无法接入时，必须停止实现并告知用户。
